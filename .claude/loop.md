# Boss 直聘日间自动招聘 Loop

本 Loop 由用户在项目根目录手动运行 `/loop 1m` 启动。该手动调用授权本会话在下述时间、资格、额度和风险边界内执行真实 Boss 招聘动作，无需逐候选人确认。`1m` 是约一分钟的唤醒节奏，不是单轮超时；调度抖动或长回合可能让实际间隔略长。

## Windows + Claude Code 启动约定

本提示词专门支持 Windows 上从仓库根目录启动 Claude Code 的场景。不要把 macOS 的绝对路径带入 Windows 运行。

1. 用户必须先在包含 `.claude/loop.md` 的项目根目录启动 `claude`，再输入 `/loop 1m`。
2. 项目级 `.claude/skills/boss-zhaopin/SKILL.md` 只是桥接文件；实际业务规则从仓库相对路径 `skills/boss-zhaopin/` 加载。
3. Windows 使用 PowerShell 和 `py -3` 调用 Python；macOS/Linux 才使用 `python3`。`boss` 必须从 `PATH` 调用。
4. 状态目录优先读取环境变量 `BOSS_ZHAOPIN_STATE_DIR`，未设置时使用 Windows 的 `%USERPROFILE%\.codex\state\boss-zhaopin`。小时报告只写入该目录，不写入仓库。
5. 禁止使用 `/Users/yuyu/`、`/opt/homebrew/`、`/tmp/...` 等机器专属路径，也不要假定当前盘符、Shell 或用户名。
6. 需要定位仓库或 skill 文件时，以 Claude Code 当前项目根目录和当前 `SKILL.md` 所在目录为准；不要 `cd` 到硬编码目录。

## 权威来源与平台

1. 加载项目 skill `.claude/skills/boss-zhaopin/SKILL.md`，并以 `skills/boss-zhaopin/` 中的业务规则和 references 为唯一业务来源。
2. 不使用 `boss-recruiting-agent/` 的历史规则，不猜测话术，不自动执行 `boss update`。
3. 从当前 `SKILL.md` 位置解析 skill 根目录，不硬编码盘符、用户名或 macOS 路径。
4. Windows 优先使用 `py -3`；macOS/Linux 使用 `python3`。`boss` 必须从 `PATH` 调用。
5. 本任务是单执行器。Claude Code 的下一次定时提示即使到期，也必须等当前回合结束；忙碌期间错过多次触发时只补执行一次，不得补跑多个旧轮次。
6. 不得启动子进程、子代理、第二个 Claude 会话或另一台电脑并行操作 Boss。发现其他招聘执行器可能仍在运行时立即暂停。

## 运行时间

使用 Asia/Shanghai（UTC+8）。只在当天 09:00（含）至 21:00（不含）发送消息或打招呼。

- 当前时间早于09:00或已达到21:00：不执行招聘动作，保存报告，取消当前 Loop 并结束。
- 每个状态变更动作前重新检查时间。预计不能在21:00前完成完整候选人动作时，不开始该动作。
- 21:00后只允许完成已成功动作的只读验证和本地记录。
- 本次 Loop 不跨天；第二天由用户重新手动运行 `/loop 1m`。

## 首轮启动

本会话第一次运行时：

1. 解析 `skills/boss-zhaopin/scripts/runtime_store.py`。
2. 执行 `init` 和 `purge`，查询当日 `greeting-count` 与 `due-followups`。
3. 执行 `boss help`，再用 `boss list --unread` 验证登录。
4. 需要登录时执行 `boss login`，提示用户完成登录；成功前不执行后续动作。
5. 在状态根目录创建本次会话元数据。状态根目录优先取 `BOSS_ZHAOPIN_STATE_DIR`，否则使用用户主目录下 `.codex/state/boss-zhaopin`。
6. 小时报告写入 `<状态根目录>/hourly-reports/YYYY-MM-DD/<启动时间>/`，不得写入 Git 仓库。

后续轮次复用本地会话元数据，不重复创建另一个执行器。

## 每轮顺序

### 1. 清空未读

反复执行 `boss list --unread`。逐名处理，不并行：

1. 用 `boss chat "<姓名>" --strict` 精确打开会话并读取完整可见对话。
2. 回复前重新执行资格门槛：2027届、本科或研究生、最终学历学校命中目标名单、专业语义符合正式规则。任何字段不符合或无法确认都不回复，也不暴露筛选原因。
3. 按正式状态机回复每条新消息。每次发送、接收简历、交换微信或终止后重新读取会话验证，再记录本地事件。
4. 检测到用户手动发送的新消息时标记 `manual_takeover`；候选人再次回复前不执行旧计划。
5. 每处理完一人立即重新查询未读，直到当前未读为零。不得依赖本轮开始时的旧列表。

### 2. 处理到期跟进

查询达到六小时的跟进。每次只处理一名，发送前重新打开精确会话并验证计划仍有效。成功后才记录次数，最多八次。

每处理一名跟进，立即返回“清空未读”；未读清空后再处理下一名到期跟进，直到当前没有到期任务。

### 3. 新增一名招呼

重新查询当天成功招呼数。

- 少于150：按推荐、资格、批次、长期去重和安全规则只招呼一名合格候选人，并完整执行真人化说明、合并岗位介绍、附件简历请求、验证和状态记录。本轮不新增第二名。
- 达到150：不进入推荐页，不执行 `boss greet`。仍继续处理未读和到期六小时跟进。
- 任何 `greet`、`chat`、`send` 或状态动作失败后，不盲目执行下一步。

完成一名候选人的整套动作后立即重新检查未读，然后结束本轮，等待下一次定时提示。

## 知识缺口

不回答、不猜测知识库没有明确答案的问题，只记录脱敏摘要。

只有候选人通过资格门槛且 `evaluate-transfer` 明确返回 `exchange_wechat` 时，才按 skill 的“知识缺口转微信”分支发送批准话术、执行 `boss action wechat` 并验证。不得用知识缺口绕过投递部门、心理测评、面试状态、拒绝、停止联系、投诉、举报或平台风险规则。

## 小时报告

从本次手动启动时刻起每满60分钟，在当前原子动作完成后的第一个安全检查点生成一份 Markdown 小时报；21:00再生成最终报告。

报告至少包含报告区间、当日累计招呼及剩余额度、本小时未读处理、成功回复、到期跟进、简历、投递状态、微信交换、停止原因、知识缺口和异常。只使用本地真实事件；缺失数据写“无记录”或“未知”，不得推断。不得写入简历正文、手机号、微信号、Cookie、Token或完整聊天记录。

空闲轮次保持简短，不要每分钟向用户重复“没有未读”。

## 必须暂停整个 Loop

验证码、平台风控或异常弹窗、投诉/举报/法律风险、登录无法恢复、CLI连续失败、状态变更无法验证、本地状态损坏，或发现可能存在并行发送时，立即停止所有新动作，保存脱敏错误和报告，取消当前 Loop 并通知用户。

普通资料不全或不能安全继续的单个候选人只暂停该候选人，不阻塞其他人。
