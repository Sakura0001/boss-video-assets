# 自动回复

1. 使用 `boss list --unread` 获取未读会话。
2. 逐个执行 `boss chat "<姓名>" --strict`，读取完整可见对话。
3. 回复前校验 2027 届、本科或研究生、页面可见全部教育经历中至少一所学校命中目标名单、当前学历对应专业语义上能明确判断为计算机类或与 `school_policy.yaml` 的 `allowed_majors` 某一方向相近；硕士或博士显示多所学校时任意一所精确命中即可。学校自定义的近似专业名称可以通过，明显无关、无法判断或其他字段不符合时不回复。
4. 检测用户手动发送的新消息；存在时标记 `manual_takeover`，等待候选人再次回复。
5. 候选人请求附件简历时按 `candidate_conversion.md` 接收并验证。
6. 候选人问机会时使用 `followups.md` 的固定首问或重复问分支。
7. 候选人询问投递链接或招聘/秋招是否开始时，使用 `application.md` 中对应岗位的官网、步骤和固定回复；方向不明时先问软件开发/测试、AI 或网络安全方向，随后发送微信交换请求。
8. 候选人谈投递、转投、测评、笔试或面试时，提取结构化字段并调用 `evaluate-transfer`。
9. 候选人询问某个 base 是否有 HC，或某个 base 是否有某个岗位/职位时，统一回复：`有`。候选人追问 HC 多不多时，统一回复：`还可以，和往年持平，有几十个`。
10. 候选人表示“还没准备好”、正在准备材料或类似犹豫时，统一回复：`可以先投递，先投递先进流程，性格测试或者笔试没过还可以申请二次测试，投递晚了就申请不了了。`
11. 其他未覆盖的 offer、岗位名额、团队安排或录用细节不得编造；不发送，记录脱敏问题到 `offer_questions_pending.md`，并提示用户统一补充口径。
12. 普通问题只使用已安装 references 中明确的答案。
13. 知识库没有答案时不回答未知问题；记录 `knowledge_gap` 的脱敏摘要。只有候选人通过资格门槛且 `evaluate-transfer` 返回 `exchange_wechat` 时，才按 `candidate_conversion.md` 的“知识缺口转微信”分支继续；否则暂停该候选人并提示用户。
14. 每次成功动作后记录对应事件，失败动作不计数。

交换微信、附件接收、投递终止等状态动作后必须重新读取会话验证。
