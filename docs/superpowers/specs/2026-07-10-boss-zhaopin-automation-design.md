# Boss 直聘自动招聘 Skill 设计规范

## 1. 目标与成功标准

本次改造将 `boss-zhaopin` 从话术参考 skill 升级为按需运行的自动招聘 skill。用户显式调用 skill 后，系统检查 Boss 登录状态；登录完成后自动执行当次招聘流程，不逐条请求发送确认。

目标候选人是符合学校、专业、学历和届别要求，且满足以下任一条件的人：

- 尚未投递华为。
- 已投递华为，但未投递 ICT 下云软件研发部，且尚未参加过面试。
- 已投递其他部门且笔试未通过，但心理测评未失败、尚未参加过面试。

成功事件定义为 `wechat_exchange_completed`：候选人或招聘方任一方发起交换微信，且 Boss 页面显示微信交换动作已经完成。完成交换即计为成功，不要求系统验证微信客户端中是否已经添加好友。

## 2. 方案选择

### 方案 A：只维护流程文档

由 Codex 每次临时理解 Markdown 并执行 Boss CLI。优点是改动小，缺点是无法可靠去重、计算六小时间隔、记录追问次数或生成日报。当前项目基本属于此方案，不满足目标。

### 方案 B：Skill 驱动 + 确定性本地状态脚本（采用）

由 skill 负责筛选、自然语言理解和 Boss CLI 编排；由脚本负责状态、时间、去重、台账和日报。每个真实发送动作仍逐个执行并校验，不使用 shell 循环批量发送。

该方案既能利用模型理解候选人自然语言，又能让关键计数和状态具有确定性，是本次推荐方案。

### 方案 C：独立常驻招聘守护进程

由后台程序持续轮询并自动发送。优点是无需手动启动，缺点是浏览器状态、验证码、平台风控、自然语言分支和人工接管更难控制。用户已明确采用“启动 skill 时运行”，因此本次不实现常驻进程。

## 3. 唯一事实来源与发布

- 唯一事实来源为 `/Users/yuyu/.codex/skills/boss-zhaopin/`。
- 运行时只读取已安装 skill，不从 `boss-recruiting-agent/` 获取业务口径。
- 仓库中的 `skills/boss-zhaopin/` 是已安装 skill 的同步副本，不允许独立修改业务规则。
- 每次用户确认新口径后，先更新已安装 skill，运行验证，再同步仓库副本、提交并推送 GitHub。
- GitHub 推送失败时明确提示，不把未推送状态报告为成功。
- `boss-recruiting-agent/` 降级为历史资料，不再作为运行时规则来源；项目入口说明应改为指向已安装 skill。

## 4. 调用与运行模式

用户调用 `$boss-zhaopin` 并要求自动招人时：

1. 检查 `boss` CLI 和 Chrome/CDP 可用性。
2. 检查 Boss 登录状态；未登录时执行 `boss login`，等待用户完成登录后继续。
3. 清理超过三天的临时状态。
4. 扫描全部未读会话，先做资格校验，再决定是否回复。
5. 处理已到期的六小时跟进任务。
6. 在每日剩余额度内处理推荐候选人，主动打招呼每日最多 150 人。
7. 对当前会话逐个推进到等待候选人、终止状态或微信交换完成。
8. 更新本地状态、长期微信台账和当日日报。
9. 输出本次执行摘要和需要用户补充的未知知识。

skill 不是后台定时任务。六小时跟进只在用户再次运行 skill 时检查；距离上次发送已满六小时的候选人才进入待跟进队列。

## 5. 候选人资格

候选人必须同时满足：

- 2027 年毕业，属于 2027 届。
- 学历为本科或研究生。
- 读取页面右侧显示的全部结构化教育经历学校。
- 本科通常显示一所，硕士或博士可能显示多所；任意一所学校精确命中
  `references/target_schools.md` 即通过学校门槛。
- 专业属于理工科；文科专业不符合。
- 不要求编程、软件、数据库、测试或工作经历，可以从零培养。

学校、专业、学历或毕业年份任一不符合或无法识别时，不主动打招呼，也不回复主动私信，不向候选人解释内部筛选规则。

## 6. 新候选人触达

1. 使用 `boss recommend <岗位关键字>` 获取候选人。
2. 完成资格校验和长期去重校验。
3. 执行 `boss greet "<姓名>" --job <岗位关键字>`。
4. 使用 `boss chat "<姓名>" --strict` 打开精确会话。
5. 发送真人化补充：

   `我看了下你的背景，和我们数据库方向挺匹配的，方便的话我给你简单介绍下我们这边方向。`

6. 发送一条合并岗位介绍：

   `我这边主要是华为数据库业务相关团队，方向包括TaurusDB、RDS、GaussDB、GeminiDB、Redis等，岗位主要偏数据库内核和管理系统研发和测试。数据库业务是公司重点投入方向，目前也在拓展海外市场，后续业务空间比较大。这个领域本身比较垂直，技术积累会比较深，长期做下来也更容易形成自己的技术护城河。`

7. 索要附件简历：

   `方便的话辛苦发我一份附件简历，我帮你进一步看下和数据库方向的匹配度。`

开场固定事实、投递规则、机会回答和微信交换话术不做随机改写。未回复跟进可以使用多个不改变事实的自然表达版本，并记录使用版本。

## 7. 简历与投递状态

候选人发起附件简历请求后：

1. 发送：`收到同学，我先看下你的简历，稍等一下。`
2. 执行 `boss action agree-resume`。
3. 重新打开精确会话，确认 `简历获取状态: 已获取`。
4. 简短等待后发送：

   `我看了下，你的简历和我们这边方向挺匹配的。同学你这边是不是还没有投递呀？`

5. 等待候选人回答，不在投递状态确认前交换微信。

如果候选人表示投递过、询问投递过是否还能投，继续询问：

`同学你投的哪个部门呀？简历编号有给过别人吗？`

如判断转投资格还需要流程阶段，继续自然确认心理测评结果及是否参加过面试。简历编号是否给过别人本身不决定资格，部门和流程阶段才是最终条件。

候选人回避投递状态时，总共询问两次。第二次可使用：

`同学你还没说投没投递过呢，简历编号有给过别人嘛？`

两次询问后仍不回答，放弃候选人，不交换微信。

## 8. 转投资格决策

按以下优先级判断：

1. 明确投递过 ICT 下云软件研发部：礼貌说明无法再次投递，终止联系。
2. 心理测评未通过：不能再次投递，终止联系。
3. 已参加过任一轮面试，无论结果如何：不能转投，终止联系。
4. 尚未参加面试且投递部门不是云软件研发部：可以转投，进入微信交换。
5. 其他部门笔试未通过，但心理测评未失败且未参加面试：可以投递我们部门重新笔试，无等待期，进入微信交换。
6. 只知道投过华为、说不清组织：进入微信交换，后续在微信确认。
7. 尚未投递：进入微信交换。

其他部门示例包括终端、终端云、车 BU、无线、计算产品线等。系统不根据示例自行扩大“云软件研发部”的终止范围；只有明确命中云软件研发部时才应用该终止原因。

标准招聘流程口径：

`投递 → 心理测评 → 笔试 → 两轮技术面试 → 一轮主管面试 → 提交审批 → 发 offer`

## 9. 微信交换

满足资格后执行：

1. 发送：

   `同学，沟通下来我感觉你和我们数据库方向还挺匹配的。如果你方便的话，可以加我微信，后续沟通会更及时一些，我也可以继续帮你跟一下流程。`

2. 执行 `boss action wechat`。
3. 发送：

   `辛苦你这边主动加我一下哈，我们账号一天不能主动添加太多同学，频繁添加容易异常。加上后你备注一下姓名，我后续在微信上跟你继续沟通。`

4. 重新读取会话并确认交换动作完成。
5. 将候选人标记为 `wechat_exchange_completed`，写入长期台账并停止所有主动跟进。

交换完成后候选人仍在 Boss 提问时，只说明后续投递和流程在微信跟踪，不在 Boss 重复展开。

## 10. 固定问答口径

### 机会大小

首次询问：

`我看了下机会挺大的，只要你性格测评和笔试通过了，这个帮不了你，到后面面评不差的话offer概率还是很大的`

重复询问或质疑：

`没骗你哈，我不会拿这个忽悠你。你这个背景确实可以往我们方向推，但测评、笔试和面试还是要自己过。我们先加个微信，后续投递和编号我在微信上跟你说。`

以上是用户批准的固定业务口径，不再被“录用概率统一人工审核”的旧规则拦截；仍禁止扩写成“一定录用”或“一定发 offer”。

### 薪资

`薪资主要根据面评结果确定，后面可以再沟通，现阶段还是先把笔试和面试通过。`

不自动提供范围或具体金额。

### 城市

可选城市为北京、西安、上海、成都、东莞、深圳、南京、杭州，候选人可以选择。

### 作息

所有岗位通常周一、周二、周四约 20:30 下班，周三、周五约 17:30 下班。通常每月有一两次会因业务忙到 22 点以后，实际节奏会随业务变化，不表述为绝对保证，也不攻击其他部门。

### 海外协作

所有岗位都会参与海外研究团队的线上协作，可能存在时差，具体频率随项目安排。可以表述为日常存在海外联合研发和跨团队沟通机会，不承诺固定次数。

### 深夜在线

候选人问为什么这个时间还在上班时，可回复：

`下班了刷刷boss招人，团队需要的人比较多。`

### 未知问题

知识库没有明确答案时不自动发送。整理候选人问题、必要上下文和建议草稿并提示用户处理。用户确认答案后，先更新已安装 skill，再同步仓库副本。

## 11. 跟进与停止条件

候选人在初始岗位介绍和索要简历后完全不回复：

- 每隔六小时允许一次主动跟进。
- 最多跟进八次。
- 八次话术使用不同自然表达，不重复同一句，不增加未经确认的业务事实。
- 候选人明确拒绝、不考虑、要求停止联系、投诉或举报时立即终止，不再执行剩余跟进。
- 候选人回复后取消旧的定时跟进计划，按最新会话重新判断。

检测到用户手动发送的新消息后，暂停该候选人的自动跟进。候选人再次回复时，重新读取完整对话并从最新状态继续，不能补发旧计划消息。

## 12. 状态机

主要状态：

- `eligible`
- `greeted`
- `waiting_resume`
- `resume_received`
- `waiting_application_status`
- `handoff_eligible`
- `wechat_exchange_completed`
- `manual_takeover`
- `stopped`

主要终止原因：

- `ineligible_or_unknown_school`
- `ineligible_or_unknown_major`
- `ineligible_or_unknown_graduation_year`
- `not_2027_cohort`
- `already_cloud_software_department`
- `psychological_assessment_failed`
- `interview_already_started`
- `candidate_declined`
- `candidate_requested_stop`
- `max_followups_reached`
- `application_status_unanswered`

每次发送前重新打开 `boss chat "<姓名>" --strict`，确认最新消息和当前状态。每次附件、微信等状态变更动作后重新读取会话验证。

## 13. 本地数据

统一状态根目录：

`/Users/yuyu/.codex/state/boss-zhaopin/`

目录权限设为仅当前用户可读写，所有文件均禁止进入 Git。

### 三天临时状态

保存匿名候选人标识、当前阶段、上次联系时间、追问次数、跟进版本和微信交换结果。三天后自动删除。

### 长期去重索引

仅保存 Boss 会话标识、首次联系时间和最终状态，保留到 `2027-12-31`。已拒绝、不符合条件、已投云软件研发部、流程终止或已交换微信的候选人不再主动联系。

### 长期微信交换台账

长期保留，供用户在微信中二次核对。每条记录包括：

- Boss 显示姓名
- Boss 会话标识
- 交换完成时间
- 命中的教育经历学校
- 专业
- 学历
- 投递状态
- 原投递部门
- 备注

不保存简历正文、手机号、微信号、Cookie、Token 或聊天全文。

### 每日日报

按日期保存并可随时组装，包含：

- 主动打招呼数
- 候选人回复数
- 收到简历数
- 确认未投递数
- 确认非云软件研发部数
- 完成微信交换数及候选人名单
- 各阶段转化率
- 停止联系人数及原因
- 知识库未覆盖问题

日报和微信交换台账长期保留，仅存在本机。

## 14. 错误处理

- Boss 登录失效：停止发送，执行登录流程，成功后重新读取状态再继续。
- 验证码、风控、异常弹窗：立即暂停所有发送并提示用户。
- `boss greet`、`boss chat` 或 `boss send` 失败：不执行后续动作，记录失败并提示。
- 无法精确打开候选人会话：不发送。
- 附件简历未确认获取：不声称已看简历，不询问投递状态。
- 微信交换动作未确认完成：不记为成功，不写长期台账。
- 知识库缺失：不编造、不发送，转人工补充。
- 手动消息与自动计划冲突：以手动消息为准，取消旧计划。

## 15. 测试与验收

实现阶段必须覆盖：

1. 本地状态脚本的测试先失败，再实现通过。
2. 三天临时状态清理和 `2027-12-31` 去重保留规则。
3. 微信交换成功只在动作确认后写入台账。
4. 同一候选人重复运行不会重复打招呼或重复交换微信。
5. 手动消息出现后暂停旧跟进。
6. 不符合或无法识别学校、专业、届别的未读候选人不产生回复动作。
7. 云软件研发部、心理测评失败、已参加面试三类终止分支。
8. 未投递、非云软件研发部未面试、其他部门笔试失败三类交换微信分支。
9. 每日 150 次主动打招呼上限。
10. 六小时跟进间隔和最多八次限制。
11. 日报可以按指定日期组装，并列出当天完成微信交换的候选人。
12. skill 结构验证、仓库副本一致性检查和敏感信息扫描。

真实 Boss 前向测试不得批量执行。先使用模拟 CLI 输出测试状态机；需要真实发送验证时逐个候选人执行并在每步后重新读取会话。

## 16. 非目标

- 不实现后台常驻轮询服务。
- 不接管微信端后续招聘流程。
- 不维护投递链接或真实 URL。
- 不保存候选人简历正文、手机号、微信号或完整聊天记录。
- 不回复资格不符合或资料无法识别的主动私信候选人。
- 不对知识库外的信息作出自动承诺。
